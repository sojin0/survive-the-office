## 1. 핵심 디자인 철학

**"감정은 색이 아니라 날씨로 말한다."**

스트레스와 회복을 숫자나 색상 막대로 표현하는 기존 무드 트래커와 달리, 이 서비스는 **날씨**라는 친숙한 메타포를 사용합니다. 날씨는 설명 없이도 오늘의 상태를 직관적으로 전달하며, 팀과 공유할 때도 부담이 없습니다.

---

## 2. 색상 시스템 선택 근거

### 2.1 모노톤 베이스 컬러

**결정**: UI의 기본 색상을 흑백 모노톤(`#1A1A1A` ~ `#FFFFFF`)으로 구성

**근거**:
- 날씨 테마 배경색(노랑, 파랑, 회색 계열)이 강하기 때문에 UI 요소가 경쟁하면 시각적으로 복잡해짐
- 모노톤 베이스는 어떤 날씨 배경에도 충돌 없이 자연스럽게 어우러짐
- 감정 상태를 UI가 아닌 **배경 날씨**가 전달하도록 역할을 분리

**고려했던 대안**: 브랜드 컬러 기반 컬러풀한 UI → 날씨 배경과 충돌하여 기각

---

### 2.2 날씨별 배경 그라데이션

**결정**: HP 범위에 따라 6단계 배경 그라데이션이 전체 페이지에 적용

| 날씨 | 색상 | 선택 이유 |
|---|---|---|
| ☀️ 맑음 | 노란색 계열 | 햇살, 에너지, 긍정의 보편적 연상 |
| ⛅ 구름 조금 | 하늘색 계열 | 맑지만 약간의 여운 |
| ☁️ 흐림 | 회색 계열 | 중립적, 무난한 하루 |
| 🌧️ 비 | 남색 계열 | 무겁고 우울한 감정의 시각화 |
| ⛈️ 폭풍 | 어두운 청회색 | 극도의 피로, 위기감 |
| 💀 전멸 | 블랙 계열 | 완전한 소진, 게임 오버 |

**구현 방식**: CSS 클래스(`.weather-sunny` 등)를 최상위 컨테이너에 적용하여 CSS 변수를 하위 전체에 전파 → 텍스트·버튼·카드 색상이 날씨에 따라 자동 전환

---

### 2.3 다크 날씨 대응

**결정**: stormy/dead 상태에서 텍스트·버튼·툴팁 색상을 흰색 계열로 자동 전환

**근거**: 어두운 배경에서 기본 검정 텍스트는 가독성이 심각하게 떨어짐. CSS 변수 재정의 방식으로 컴포넌트 코드 변경 없이 테마 전환 구현

---

## 3. HP 게이지 + 날씨 메타포 선택 이유

### 3.1 왜 HP(체력 게이지)인가?

**문제**: 스트레스 수치, 피로도, 감정 점수 등 다양한 표현 방식 검토

**결정**: RPG 게임의 HP 시스템 차용

**근거**:
- HP는 직관적으로 "남은 에너지"를 의미하며 별도 설명이 필요 없음
- 게임적 요소가 진지할 수 있는 번아웃 주제를 **가볍고 유쾌하게** 접근하게 함
- 이벤트로 HP가 변하는 구조가 인과관계를 명확히 보여줌
- 생존 등급(S~F)으로 이어지는 게임 루프가 재참여 동기를 만듦

### 3.2 초기 HP를 80으로 설정한 이유

**초기 설계**: HP 100에서 시작  
**변경**: HP 80으로 조정

**근거**: 사용성 테스트에서 "출근 자체가 이미 체력 소모인데 100은 비현실적"이라는 피드백 다수. 80 시작은 출근·준비 과정의 기본 피로도를 반영하며, 긍정 이벤트(커피 한 잔 +8)를 눌렀을 때 HP가 오르는 것이 시각적으로 의미 있게 보임

### 3.3 왜 날씨 메타포인가?

**근거**:
- 날씨는 **설명 없이** 감정 상태를 전달하는 보편적 언어
- "오늘 날씨 어때요?"는 "오늘 스트레스 몇 점이에요?"보다 훨씬 자연스러운 소통
- 팀 공유 시 수치보다 날씨 이모지가 심리적 부담이 낮음

## 4. 컴포넌트별 디자인 결정 사항

### 4.1 EventButton — 이벤트 버튼

**결정**: 긍정/부정 탭 분리 + 2×N 그리드

**근거**:
- 긍정/부정을 한 화면에 모두 보여주면 버튼 수가 많아 시각적으로 혼잡
- 탭 분리로 현재 맥락에 집중할 수 있게 함
- 2열 그리드는 모바일에서 엄지 닿기 편한 크기 확보

**툴팁 설계**: 클릭 즉시 이벤트 설명이 버튼 위에 나타났다 사라지는 피드백 → "내가 무언가를 했다"는 즉각적 반응이 기록 행동을 강화

### 4.2 HPGauge — HP 게이지

**결정**: 수치 + 상태 뱃지 + 바 + 범위 힌트 + 토스트 메시지 구조

**근거**:
- 수치만으로는 맥락이 없어 "최상/양호/주의" 등 상태 텍스트 추가
- 0~100 범위 힌트(0·50·100)로 현재 위치 직관적 파악
- 이벤트 클릭 시 토스트 메시지("소소한 행복! ☕")로 감정적 피드백 강화
- HP 극단값(0, 100) 고정 메시지로 특별한 순간 강조

### 4.3 SurvivalResult — 퇴근 결과 화면

**결정**: 등급별 차별화된 연출 (S/A: 파티클 + 빛나는 링 / F: 흔들림 애니메이션)

**근거**:
- 단조로운 결과 화면은 퇴근이라는 하루의 클라이맥스를 살리지 못함
- 좋은 날은 화려하게 축하하고, 힘든 날은 위로하는 감성적 연출로 차별화
- HP 카운트업 애니메이션으로 결과를 "발표받는" 긴장감 연출

### 4.4 WeatherCard — 날씨 카드

**결정**: 컴포넌트 자체에 `weather-${state}` 클래스 적용

**근거**: 초기에는 날씨 클래스를 상위 컨테이너에만 적용했으나, WeatherCard가 독립적으로 렌더링될 때 테마가 적용되지 않는 문제 발생. 컴포넌트 자체에 클래스를 적용해 자급자족적 테마 전환 구현

### 4.5 TimelinePanel — 타임라인

**결정**: 세로 라인을 이벤트 목록 높이에만 제한, 팀원 응원 섹션과 분리

**근거**: 세로 라인이 응원 섹션까지 침범하면 시각적으로 어색함. ref로 실제 DOM 높이를 측정해 라인 길이를 동적으로 제한

## 5. 레이아웃 설계 결정

### 5.1 대시보드 2컬럼 레이아웃

**결정**: 왼쪽 메인(내부 스크롤) + 오른쪽 타임라인(고정 sticky)

**근거**:
- 이벤트를 기록하면서 동시에 타임라인을 확인하는 사용 패턴
- 타임라인이 페이지 스크롤과 함께 사라지면 기록 확인이 불편
- 왼쪽 영역만 스크롤되어 타임라인은 항상 노출

### 5.2 모바일 우선 설계

**결정**: 모바일에서 단일 컬럼, 하단 탭 네비게이션

**근거**: 서비스 특성상 업무 중 모바일로 빠르게 기록하는 시나리오가 주요 사용 패턴. 데스크탑은 부가적 환경으로 설계

## 6. 반복 개선 과정

사용성 테스트(내부 3명) 및 디자인 검수 후 변경된 주요 사항들.

| 항목 | 변경 전 | 변경 후 | 계기 |
|---|---|---|---|
| 초기 HP | 100 | 80 | 테스트 참여자 피드백 — "출근 자체가 피로" |
| 툴팁 지속시간 | 2000ms | 1200ms | 직접 사용 시 너무 길게 느껴짐 |
| HistoryCalendar | 모바일 좌우 분할 고정 | 모바일 세로 스택 전환 | 테스트 P-02, P-03 오류 발생 |
| 퇴근 후 화면 | "내일 다시 시작하기" 버튼 | "대시보드 보기" + 토스트 메시지 | 퇴근 후 오늘 기록 열람 니즈 |
| 날씨 테마 텍스트 | stormy/dead에서 검정 텍스트 | 흰색으로 자동 전환 | 디자인 검수 중 가독성 문제 발견 |
| 툴팁 색상 | 항상 어두운 배경 | 날씨 테마 연동 (dark ↔ light) | stormy/dead에서 흰 툴팁 + 흰 텍스트 불가 |
| WeatherCard 테두리 | border 없음 | stormy/dead에서 outline 추가 | 어두운 배경에서 카드 경계 불명확 |
| 타임라인 세로 라인 | bottom: 0 고정 | 이벤트 목록 높이만큼만 | 팀원 응원 섹션 침범 문제 |
