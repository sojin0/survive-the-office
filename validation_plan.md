## 2. 사용성 테스트 계획

### 2.1 테스트 방법

- **방식**: 모더레이티드 테스트 (진행자 1인 동석, 화면 녹화)
- **환경**: 데스크탑 Chrome 브라우저 / 모바일 Safari (각 절반씩)
- **소요 시간**: 1인당 약 20~30분
- **측정 도구**: 화면 녹화 + 구두 Think-Aloud + 종료 후 SUS 설문

### 2.2 테스트 태스크 및 시나리오

#### 시나리오 A — 하루 시작 & 이벤트 기록

> "오늘 출근해서 서비스에 처음 접속했습니다. 오전에 커피를 마시고, 갑작스러운 긴급 요청이 들어왔습니다. 이 두 가지를 기록해보세요."

| 태스크 ID | 태스크 | 성공 기준 | 목표 완료율 |
|---|---|---|---|
| T-01 | 서비스 접속 후 현재 HP와 날씨 상태 확인 | HP 수치와 날씨 아이콘을 5초 내 인지 | 100% |
| T-02 | "커피 한 잔" 긍정 이벤트 기록 | 버튼 클릭 후 HP 증가 및 툴팁 확인 | 90% |
| T-03 | "긴급 요청" 부정 이벤트 기록 | 버튼 클릭 후 HP 감소 및 툴팁 확인 | 90% |
| T-04 | 오늘의 이벤트 타임라인에서 기록 확인 | 두 이벤트가 타임라인에 순서대로 표시됨을 확인 | 85% |

#### 시나리오 B — 퇴근 결과 확인 & 팀 공유

> "오늘 하루가 끝났습니다. 퇴근 결과를 확인하고, 팀 보드에 오늘 상태를 공유해보세요."

| 태스크 ID | 태스크 | 성공 기준 | 목표 완료율 |
|---|---|---|---|
| T-05 | "퇴근하기" 버튼 찾아서 클릭 | 오류 없이 퇴근 결과 화면 진입 | 90% |
| T-06 | 생존 등급과 오늘의 요약 내용 파악 | 등급, 타이틀, 이벤트 TOP 3 인지 | 85% |
| T-07 | 팀 보드에 오늘 결과 공유 | 팀 보드에 본인 상태가 업데이트됨 확인 | 80% |

#### 시나리오 C — 히스토리 조회

> "지난주 가장 힘들었던 날의 기록을 찾아보세요."

| 태스크 ID | 태스크 | 성공 기준 | 목표 완료율 |
|---|---|---|---|
| T-08 | 히스토리 캘린더 진입 | 메인에서 히스토리로 2단계 이내 진입 | 85% |

### 2.3 측정 지표

| 지표 | 측정 방법 | 목표값 |
|---|---|---|
| 태스크 완료율 | 성공/실패 이분법 기록 | 전체 평균 85% 이상 |
| 태스크 소요 시간 | 초 단위 측정 | T-02, T-03 각 10초 이내 |

| SUS 점수 | 10문항 설문 (0~100점) | 70점 이상 (Good 등급) |
| NPS | 단일 문항 (0~10점 추천 의향) | 7점 이상 |

### 2.4 SUS 설문지

> 각 문항: 1(전혀 그렇지 않다) ~ 5(매우 그렇다)

1. 이 서비스를 자주 사용하고 싶다
2. 이 서비스는 불필요하게 복잡하다
3. 이 서비스는 사용하기 쉽다
4. 이 서비스를 사용하려면 기술 전문가의 도움이 필요할 것 같다
5. 이 서비스의 다양한 기능이 잘 통합되어 있다
6. 이 서비스에는 일관성이 없는 부분이 너무 많다
7. 대부분의 사람들이 이 서비스를 빠르게 배울 수 있을 것이다
8. 이 서비스는 사용하기 매우 불편하다
9. 이 서비스를 사용하는 데 자신감을 느꼈다
10. 이 서비스를 사용하기 전에 많은 것을 배워야 했다

## 4. A/B 테스트 계획

### 4.1 A/B 테스트 #1 — 이벤트 패널 레이아웃

**가설:** 긍정/부정 이벤트를 탭으로 분리하는 것보다 한 화면에 모두 보여주는 것이 기록 속도를 높일 것이다.

| 구분 | 설명 |
|---|---|
| **A안 (Control)** | 긍정 / 부정 탭 분리형 — 탭 전환 후 이벤트 선택 |
| **B안 (Variant)** | 긍정(상단) + 부정(하단) 단일 스크롤 화면 |
| **측정 지표** | 이벤트 기록까지 소요 시간 / 오류 클릭 횟수 |
| **성공 기준** | B안이 평균 소요 시간 20% 이상 단축 시 채택 |
| **테스트 대상** | 외부 사용자 3명 (A안 1~2명 / B안 1~2명 교차) |

### 4.2 A/B 테스트 #2 — 툴팁 노출 위치

**가설:** 툴팁이 버튼 위에 뜨는 것보다 화면 중앙 상단에 토스트 형태로 뜨는 것이 더 잘 인지될 것이다.

| 구분 | 설명 |
|---|---|
| **A안 (Control)** | 버튼 바로 위에 툴팁 등장 |
| **B안 (Variant)** | 화면 상단 중앙에 토스트 메시지로 등장 |
| **측정 지표** | 툴팁 인지 여부 (인터뷰 확인) / 사용자 만족도 |
| **성공 기준** | 인지율 80% 이상인 안으로 채택 |
| **테스트 대상** | 내부+외부 혼합 4명 |

## 5. 코드 검수 체크리스트

### 5.1 CLAUDE.md 원칙 준수 여부

| 원칙 | 확인 항목 | 통과 여부 |
|---|---|---|
| 디자인 토큰 우선 | 색상/간격/반경 하드코딩 없음 | ☐ |
| 타입 안전성 | `any` 타입 사용 없음, 모든 props 타입 정의 | ☐ |
| HP 클램프 | `Math.min(100, Math.max(0, hp + delta))` 적용 확인 | ☐ |
| 애니메이션 일관성 | Framer Motion 단일 사용, CSS transition 혼용 없음 | ☐ |
| 단일 책임 | 컴포넌트별 역할 분리 확인 | ☐ |
| 접근성 코드 | aria-label, 포커스 스타일 코드 레벨 확인 | ☐ |

### 5.2 데이터 구조 검수

| 항목 | 확인 기준 | 통과 여부 |
|---|---|---|
| EventItem 타입 | id / name / emoji / description / hpDelta / eventType 필드 모두 존재 | ☐ |
| EventLog 타입 | timestamp가 "HH:MM" 형식으로 자동 생성됨 | ☐ |
| AppState 타입 | hp / weatherState / eventLog / isRetired / survivalGrade 구조 준수 | ☐ |
| getWeatherState() | HP 범위별 6가지 반환값 정확성 | ☐ |
| getSurvivalGrade() | HP 잔량별 S~F 등급 반환값 정확성 | ☐ |

### 5.3 인터랙션 동작 검수

| 항목 | 확인 기준 | 통과 여부 |
|---|---|---|
| 툴팁 등장 타이밍 | fade-in 200ms | ☐ |
